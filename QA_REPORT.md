# Отчет QA и Code Review WAG System

**Версия:** 2.0 (Детальный аудит)
**Дата проверки:** 26 марта 2026 г.

В ходе тщательной проверки структуры проекта, компонентов React и серверных операций (Server Actions) было найдено значительное количество "мусорного" кода, антипаттернов производительности и проблем с безопасностью/типизацией.

---

## 🛑 1. Критические проблемы (Срочно в работу)

### Мусорные ререндеры и утечки памяти (React Hooks)
- **Отсутствующие зависимости в useEffect / useCallback**: Линтер сообщает о 61 ошибке `react-hooks/exhaustive-deps`. Разработчики часто игнорируют массив зависимостей, что приводит либо к бесконечным ререндерам (утечки памяти), либо к использованию устаревших замыканий (stale closures) при обновлении данных.
- **Ошибочное использование `createClient()` (Supabase) в теле компонентов**: в некоторых компонентах `supabase` создавался заново при каждом рендере (хотя частично поправлено кэшированием или `useRef`, нужно пройтись по всем хукам, чтобы удостовериться, что `createClient` не вызывает повторные подписки каналов на каждый чих).

### Обход безопасности через типизацию `any`
Разработчики массово используют `any`, полностью аннулируя защиту TypeScript:
- В `src/lib/actions/log.ts` — клиент supabase протипизирован как `any`.
- В `DailyReportClient.tsx` (и других `Client` компонентах) пропсы `teamReports`, `teamMembers` и `author` получают тип `any[]`. Если структура ответа базы данных изменится, проект упадёт в рантайме без предупреждения на этапе билда.

## ⚠️ 2. Средние проблемы производительности ("Тяжелый" код)

### Мега-компоненты (God Objects)
Проект содержит слишком большие файлы, смешивающие логику, состояние и множество внутренних UI-компонентов:
- `EventsCalendar.tsx` — свыше 1170 строк.
- `ProjectPipelineView.tsx` — свыше 1140 строк.
- `MyStagesView.tsx` — около 1100 строк.
- `DailyReportClient.tsx` — более 900 строк (внутри 4 огромных функции: ReportView, ReportForm, HistoryCard, TeamView).

**Улучшение:** Каждая из этих под-функций должна лежать в отдельном файле в папке `src/components/.../subcomponents`. Это ускорит Fast Refresh для разработчиков и разгрузит парсер.

### Ненужные запросы и N+1 Queries
- **Логирование и вставки пачками (Bulk inserts):** В `src/lib/actions/tasks.ts` и `events.ts` во время циклов `for` отправляются отдельные `await supabase.from(...).insert(...)`. Надо переписать на отправку массивов (batch insert: `supabase.from().insert([{...}, {...}])`), чтобы ускорить операцию в 10 раз и уменьшить нагрузку на БД.
- **`window.location.reload()` в Next.js App Router:** Этот грубый анти-паттерн до сих пор висит в файлах (`HandoverBoard.tsx`, `DailyReportForm.tsx`). Он заставляет браузер скачивать весь HTML/CSS/JS заново. Вместо этого нужно использовать `router.refresh()` из `next/navigation` для soft-перезагрузки данных.

## 🧹 3. "Мусорный код" и неиспользуемые переменные

Линтер сообщает о 27 предупреждениях, в основном:
- `@typescript-eslint/no-unused-vars`: Оставленные "на потом" переменные, импорты, которые просто висят и засоряют билды.
- В `CODE_REVIEW.md` оставлен вывод старого ревью, но новые проблемы (RLS политики `activity_log`, `projects` без достаточных проверок) так и не были до конца закрыты.
- Дублирование кода: Виджеты календаря (DatePicker), панели файлов (DocumentsPanel) скопированы по разным файлам вместо создания атомарных общих UI-модулей.

## 🛠 Документ для Разработчика (Task-лист)

Пожалуйста, передайте разработчикам следующие тикеты на рефакторинг:

- [ ] **ТИКЕТ 1. Очистка хуков.** Запустить `npm run lint` и исправить все 61 `exhaustive-deps` ошибки. Обернуть громоздкие функции вычислений в интерфейсах списков (такие как фильтрация задач в `AssignTaskList.tsx`) в `useMemo`.
- [ ] **ТИКЕТ 2. Ликвидация `any`.** Проверить поиск по коду `grep -r ": any" src/`. Заменить все `any` на сгенерированные типы Supabase или точные интерфейсы `type Profile = ...`.
- [ ] **ТИКЕТ 3. Удаление `window.location.reload()`.** Срочно заменить на вызовы перекэширования: `router.refresh()` или серверные вызовы `revalidatePath()`.
- [ ] **ТИКЕТ 4. Дробление файлов.** Разделить файлы > 500 строк на логические блоки (отделить API вызовы в кастомные хуки `useTasks()`, вьюшки вынести в папки).
- [ ] **ТИКЕТ 5. Оптимизация БД-запросов.** Проверить все файлы в `src/lib/actions/`. Если есть `while` или `for` в которых стоят `await supabase.insert`, переписать на передачу массива объектов в один `insert`.

> **Совет техлиду:** Добавьте strict mode и husky pre-commit хуки, запрещающие коммит при наличии `eslint` ошибок и типа `any`, чтобы больше этот мусор не накапливался.
