# План разработки модуля планирования проекта

> Скоуп: только проектирование. Строительные этапы исключены.
> Стек: Next.js (App Router) + Supabase + TypeScript

---

## Текущее состояние

В системе уже есть:
- `projects` — карточка проекта с менеджером, бюджетом, дедлайном
- `project_stages` — произвольные этапы (kanban-колонки)
- `tasks` — задачи внутри этапов
- `documents` — файлы по проекту/задаче
- `activity_log` — лог действий

**Проблема:** этапы сейчас generic и создаются вручную. Нужен фиксированный шаблон из 8 этапов проектирования с чек-листами, статусами и документами на каждом.

---

## Архитектурные решения

### 1. Тип проекта
Добавить поле `project_type` в таблицу `projects`:
- `design` — проектирование (текущий скоуп)
- `construction` — строительство (заморожено)

При создании проекта типа `design` — автоматически создаются 8 предустановленных этапов.

### 2. Расширение таблицы `project_stages`
Добавить поля:
- `stage_key` TEXT — машинный ключ этапа (`contract`, `surveys`, `initial_data`, `psd`, `approval`, `obos`, `expertise`, `final_delivery`)
- `status` ENUM (`pending`, `in_progress`, `completed`, `blocked`)
- `deadline` DATE
- `responsible_id` UUID → profiles
- `completed_at` TIMESTAMPTZ
- `notes` TEXT

### 3. Чек-листы этапов (`stage_checklist_items`)
Новая таблица для подпунктов каждого этапа (например, в этапе 3: ТУ, АПЗ, Аэронавигация и т.д.):
- `id` UUID
- `stage_id` UUID → project_stages
- `label` TEXT
- `is_required` BOOLEAN
- `is_completed` BOOLEAN
- `completed_by` UUID → profiles
- `completed_at` TIMESTAMPTZ
- `document_id` UUID → documents (прикреплённый документ-подтверждение)

## Этапы разработки

### Спринт 1 — База данных и миграции

**Задачи:**
1. Добавить `project_type` ENUM в `projects`
2. Расширить `project_stages` новыми полями
3. Создать таблицу `stage_checklist_items`
4. Написать SQL-функцию `create_design_stages(project_id)` — автосоздание 8 этапов + чек-листов при создании проекта типа `design`
5. Обновить RLS-политики для новых таблиц и полей
6. Написать триггер: при `status = 'completed'` на последнем этапе — проект переходит в `completed`

**Файлы:**
- `supabase/migrations/002_design_stages.sql`

---

### Спринт 2 — Серверная логика (Server Actions / API)

**Задачи:**
1. `actions/stages.ts` — CRUD для этапов:
   - `getProjectStages(projectId)`
   - `updateStageStatus(stageId, status)`
   - `updateStageDeadline(stageId, date)`
   - `assignStageResponsible(stageId, userId)`
2. `actions/checklist.ts` — работа с чек-листами:
   - `toggleChecklistItem(itemId, isCompleted)`
   - `attachDocumentToItem(itemId, documentId)`
3. `actions/projects.ts` — обновление: при создании проекта вызывать `create_design_stages`

---

### Спринт 3 — UI: Страница планирования проекта

**Маршрут:** `/dashboard/projects/[id]/planning`

**Компоненты:**

#### `ProjectPipelineView` — основной компонент
Горизонтальная или вертикальная шкала прогресса из 8 этапов.
Каждый этап — карточка с:
- Номером и названием
- Статусом (цветная метка: серый/синий/зелёный/красный)
- Ответственным (аватар + имя)
- Дедлайном (с подсветкой просрочки)
- Кнопкой раскрытия деталей

#### `StageDetailPanel` — панель деталей этапа (drawer/accordion)
- Список чек-листов с чекбоксами
- Кнопка прикрепить документ к пункту
- Поле заметок
- История изменений этапа (из `activity_log`)
- Кнопки: «Начать», «Завершить», «Заблокировать»

#### `StageProgressBar` — общий прогресс проекта
Процент завершённых этапов и чек-пунктов, отображается в карточке проекта.

---

### Спринт 4 — UI: Интеграция в карточку проекта

**Маршрут:** `/dashboard/projects/[id]`

**Изменения:**
1. Добавить вкладки: **Обзор | Планирование | Задачи | Документы | Лог**
2. На вкладке «Обзор» — мини-прогресс по этапам (компонент `StageProgressBar`)
3. Вкладка «Планирование» — `ProjectPipelineView`
4. Убрать generic kanban для проектов типа `design`

---

### Спринт 5 — Dashboard директора

**Маршрут:** `/dashboard` (существующий)

**Добавить:**
1. Колонку «Текущий этап» в таблицу проектов
2. Фильтр по этапу: показать все проекты на стадии «Госэкспертиза» и т.д.
3. Виджет «Просроченные этапы» — список этапов с истёкшим дедлайном
4. Уведомления (in-app): ответственный получает напоминание за 3 дня до дедлайна этапа

---

### Спринт 6 — Уведомления и автоматизация

1. Supabase Edge Function: ежедневная проверка дедлайнов этапов
2. Запись в `activity_log` при каждом изменении статуса/чек-листа
3. Email-уведомление (Supabase SMTP) при назначении ответственного на этап
4. Автоматический переход этапа в `in_progress` при старте работы над чек-листом
